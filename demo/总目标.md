TruckDrivers Agentic AI 改造计划
0. 当前判断

当前项目的目标方向没有问题：我们要做的不是继续堆规则，而是按照 agentic_ai_architecture.md 里定义的方向，把系统逐步改成真正的 Agentic AI。

目标链路是：

Observe -> Understand -> Build Facts -> Deliberate -> Act -> Verify -> Learn

核心原则是：

LLM 负责理解与权衡
Python 负责事实、接口、安全、校验

也就是说，Python 不应该替 Agent 做“策略取舍”，只应该告诉 Agent：

现在有哪些事实
有哪些候选动作
每个候选动作的收益、成本、时间、风险是什么
哪些动作物理上不可能执行
哪些动作只是可能带来 soft penalty

然后让 LLM 做最终 trade-off。这个方向和现有架构文档是一致的，文档里也明确指出当前问题是“LLM 外壳 + 规则内核”，很多关键策略仍藏在 Python 分支里。

1. 总目标
1.1 我们最终要达到的状态

目标不是单纯提高某一次公开数据成绩，而是构建一个隐藏数据也能稳定工作的 Agent。

最终目标：

1. 稳定跑完整个月仿真
2. validation_error 尽量为 0
3. 不陷入长期 wait 死循环
4. 偏好罚分可控
5. 收入稳定提升，目标向 20w+ 靠近
6. 不依赖公开数据特化
7. 不为每种偏好继续写 if-else
8. LLM 真正参与关键取舍
1.2 核心改造方向

从：

rule.kind -> Python if-else -> action

改成：

raw preference + structured facts + candidate facts -> LLM deliberation -> checked action
2. 必须遵守的改造原则
2.1 Python 可以做什么

Python 可以保留这些逻辑，因为它们是事实、物理约束或接口约束：

1. 距离计算
2. 时间计算
3. 收益计算
4. 成本计算
5. 装货时间窗是否可达
6. remove_time 是否已过
7. 货源是否当前可见
8. cargo_id 是否存在于当前 query_cargo 返回结果中
9. reposition 坐标是否合法
10. wait 时长是否合法
11. action schema 是否符合接口
12. 仿真结束前是否能完成订单
13. 当前是否处于明确 hard lock
14. 候选动作对累计预算的边际消耗
15. 已发生风险、已发生罚分、历史动作摘要

一句话：

Python 只回答：事实是什么？动作是否合法？
2.2 Python 不应该做什么

下面这些行为要逐步删除或收权：

1. 不要 if rule.kind == "home_nightly" 就强制回家
2. 不要 if rule.kind == "special_cargo" 就走专门流程
3. 不要 if rule.kind == "family_task" 就进入固定策略
4. 不要 if mission_id 包含某个名字就决定优先级
5. 不要 if wait_streak > N 就无条件接单或无条件移动
6. 不要 if budget_pressure == urgent 就永远 wait
7. 不要为某类偏好写固定 reposition 目标
8. 不要按 driver_id 写策略
9. 不要按 cargo_id 写策略
10. 不要按公开数据坐标写策略
11. 不要按公开数据日期、货源出现规律写策略
12. 不要让 SafetyGate 替 Advisor 选择策略
13. 不要让 Planner 直接过滤 soft risk 候选
14. 不要因为有 soft penalty 风险就直接禁止接单
15. 不要把所有不确定动作都改成 wait

一句话：

Python 不回答：现在应该为了哪个目标牺牲哪个目标？

这个取舍应该交给 LLM。

3. 当前最大问题
3.1 PlannerScorer 权限太大

当前最大问题之一是 PlannerScorer 不是单纯的事实计算器，而是半个策略引擎。架构文档也明确指出，它现在会做风险判断、过滤、reposition、机会价值判断，甚至可能把候选全部过滤掉，导致 LLM 没有真实选择空间。

必须改

把：

PlannerScorer

逐步改成：

CandidateFactBuilder

它以后只负责构造候选事实，不负责决定策略。

CandidateFactBuilder 应输出

每个候选动作输出类似结构：

{
  "candidate_id": "take_order_C123",
  "action_type": "take_order",
  "cargo_id": "C123",
  "facts": {
    "price": 4200,
    "pickup_deadhead_km": 12.4,
    "haul_distance_km": 180.0,
    "estimated_gross_income": 4200,
    "estimated_cost": 576,
    "estimated_net_before_penalty": 3624,
    "pickup_arrival_minute": 30120,
    "finish_minute": 30580,
    "can_reach_pickup_window": true,
    "will_finish_before_month_end": true
  },
  "preference_impacts": [
    {
      "constraint_id": "monthly_deadhead_budget",
      "severity": "soft",
      "estimated_penalty": 300,
      "marginal_overage": 2.1,
      "explanation": "This action may exceed the monthly deadhead budget by 2.1 km."
    }
  ],
  "hard_invalid_reasons": [],
  "soft_risk_reasons": [
    "May exceed monthly deadhead budget."
  ]
}
不能做

CandidateFactBuilder 不能做：

1. 不能因为 soft risk 直接删候选
2. 不能决定这个订单值不值得冒险
3. 不能决定偏好优先级
4. 不能决定现在应该 wait 还是接单
5. 不能因为某个 rule.kind 直接生成固定策略
3.2 CandidateSafetyFilter 过滤太强

当前 CandidateSafetyFilter 应该只过滤 hard invalid 候选，但不能过滤 soft risk 候选。

必须改

候选池要分三层：

valid_candidates
soft_risk_candidates
hard_invalid_candidates
规则
hard_invalid_candidates:
  只能放物理不可执行、接口非法、货源不可见、时间窗不可能赶上等候选

soft_risk_candidates:
  有偏好风险、有罚分风险、有长期预算风险，但物理上可执行的候选

valid_candidates:
  没有明显 hard invalid，也没有明显 soft risk 的候选
Advisor 必须看到

LLM Advisor 至少要看到：

1. 最好的安全候选
2. 最好的 soft risk 候选
3. 被 hard filter 的代表样本和原因
4. 当前 wait 是否真的会改善局面
禁止行为
1. 不允许所有候选有 soft risk 时直接全删
2. 不允许 soft risk 候选不进入 Advisor
3. 不允许因为预算接近耗尽就只返回 wait
4. 不允许 CandidateSafetyFilter 替 Advisor 做收益/罚分取舍
3.3 SafetyGate 混入策略

SafetyGate 是必要的，但它应该只做最终安全校验。架构文档里也明确写了：SafetyGate 可以 veto illegal actions，但不应该选择策略。

SafetyGate 应该做
1. action schema 校验
2. cargo_id 是否可见
3. cargo_id 是否可接
4. wait duration 是否合法
5. reposition 坐标是否合法
6. 当前动作是否违反明确 hard lock
7. 动作是否违反物理不可执行约束
8. 动作是否会导致仿真接口报错
SafetyGate 不应该做
1. 不应该判断 soft penalty 是否值得
2. 不应该决定哪个偏好优先
3. 不应该按 home_nightly/family_task/special_cargo 白名单放行或拒绝
4. 不应该因为不确定就改成 wait
5. 不应该静默重写 Advisor 的动作
6. 不应该根据公开数据经验决定等待多久
必须新增

SafetyGate 如果拒绝动作，必须返回结构化拒绝原因：

{
  "accepted": false,
  "rejection_type": "hard_invalid",
  "rejection_code": "cargo_not_visible",
  "rejection_reason": "The selected cargo_id is not in current visible cargo list.",
  "original_action": {
    "type": "take_order",
    "cargo_id": "C123"
  },
  "suggested_repair_context": {
    "visible_cargo_count": 85,
    "top_visible_candidates": ["C456", "C789"]
  }
}

然后下一轮 Advisor 必须看到这个拒绝原因，不能静默变成 wait。

3.4 LlmDecisionAdvisor 权限不足

当前 Advisor 已经存在，但它经常是在程序筛剩下的候选里做选择。这样不是真正的 Agentic AI。

必须改

Advisor 输入必须包含：

1. 原始偏好文本
2. 结构化偏好约束
3. 当前司机状态
4. 当前时间
5. 当前位置
6. 当前可见货源摘要
7. 候选动作事实表
8. soft risk 候选
9. hard invalid 代表样本
10. 历史动作摘要
11. wait streak
12. 当前任务进度
13. 最近 SafetyGate 拒绝原因
14. 当前累计预算状态
15. 当前问题诊断
Advisor 输出不能只返回 action

建议输出：

{
  "decision": "choose_candidate",
  "selected_candidate_id": "take_order_C123",
  "policy_mode": "budget_tradeoff",
  "accepted_risks": [
    {
      "risk": "monthly_deadhead_budget_overage",
      "reason": "The expected net profit is higher than the estimated soft penalty."
    }
  ],
  "rejected_alternatives": [
    {
      "candidate_id": "wait_60",
      "reason": "Waiting does not restore cumulative monthly budget."
    }
  ],
  "next_check_condition": "after this order finishes",
  "final_action": {
    "type": "take_order",
    "cargo_id": "C123"
  },
  "reason": "Choose this cargo because it has the best net value after known risk."
}
Advisor prompt 必须强调
1. 不要默认保护所有 soft preference
2. soft penalty 是可以权衡的，不是绝对禁止
3. hard constraint 才是绝对不能违反
4. 累计预算不会因为等待而恢复
5. 长时间 wait 本身有机会成本
6. 如果所有候选都有 soft risk，要选择最小净损失或最大净收益，而不是永远 wait
7. 如果 SafetyGate 拒绝过某动作，下次必须避免同类错误
8. 如果当前任务卡住，要明确 request_replan，而不是继续 wait

5. 模型切换计划：从 Mimo 改到 Qwen3.5-Flash
5.1 当前 reason_tokens 一直是 0 的判断

reason_tokens = 0 不一定代表模型没有推理。

可能原因有：

1. 当前 Mimo 模型本身不暴露 reasoning tokens
2. 当前 API provider 不返回 reasoning token 字段
3. 当前代码没有读取 provider 返回的 reasoning 字段
4. 当前请求没有打开 reasoning/include_reasoning 参数
5. 当前模型是非 reasoning 模式
6. 当前响应被封装后只保留 content，丢掉了 reasoning metadata

所以不能单纯用：

reason_tokens > 0

来判断 Agentic AI 是否有效。

更可靠的指标应该是：

1. LLM 调用次数
2. prompt_tokens
3. completion_tokens
4. Advisor 输出是否包含 policy_mode
5. Advisor 是否比较候选
6. Advisor 是否说明 accepted_risks
7. Advisor 是否解释 rejected_alternatives
8. SafetyGate rejection 是否进入下一轮上下文
9. 最终动作是否符合 Advisor 决策
5.2 Qwen3.5-Flash 的注意事项

Qwen3.5-Flash 适合作为高频 agent 决策模型，因为它定位是 fast / cost-effective，并且部分平台显示它支持很长上下文。阿里云 Model Studio 页面把 Qwen3.5-Flash 标为“Best for simple tasks, fast and cost-effective”，并显示最高上下文窗口可达 1,000,000 tokens；OpenRouter/第三方页面也列出 qwen/qwen3.5-flash-02-23 这类模型 ID 和 include_reasoning、response_format、tools 等参数支持情况。

但是要注意：不同 provider 对同一个模型的参数支持可能不完全一致。尤其是 reasoning 字段，不同网关可能叫法不同，返回格式也可能不同。

把模型名从 Mimo 改成 Qwen3.5-Flash。

例如如果走 OpenRouter，可能是：

LLM_BASE_URL=https://openrouter.ai/api/v1
LLM_MODEL=qwen/qwen3.5-flash-02-23
LLM_API_KEY=你的key

如果走阿里云 DashScope / Model Studio，则模型名和 endpoint 要按阿里云控制台实际名称配置。

不要把 key 写进代码。

必改 2：统一 LLMClient 返回结构

建议统一封装成：

@dataclass
class LlmResponse:
    content: str
    reasoning_content: str | None
    prompt_tokens: int
    completion_tokens: int
    reasoning_tokens: int | None
    total_tokens: int
    raw_model: str
    raw_response: dict

不要让上层模块直接依赖某个 provider 的原始字段。

必改 3：reason_tokens 不能作为强依赖

当前如果代码里有类似逻辑：

if reason_tokens == 0:
    treat_as_no_reasoning()

要删除。

改成：

if advisor_output_valid:
    accept_llm_decision()
else:
    retry_or_fallback()

判断重点应该是：

1. JSON 是否可解析
2. 是否有 selected_candidate_id
3. candidate_id 是否存在
4. final_action 是否合法
5. reason 是否非空
6. policy_mode 是否合理
必改 4：强制 JSON 输出

Qwen3.5-Flash 用于 Agent 决策时，必须尽量走结构化输出。

请求中优先使用：

{
  "response_format": {
    "type": "json_object"
  }
}

如果 provider 支持 JSON Schema，就更好。

Advisor prompt 必须明确：

Return ONLY valid JSON.
Do not include markdown.
Do not include explanations outside JSON.
必改 5：处理 Qwen 可能的 reasoning/content 格式问题

有些 Qwen3.5 部署在特定推理框架下，可能出现 reasoning 字段和 content 字段解析不稳定的问题。例如 vLLM 社区里有人报告过 Qwen3.5 模型配合 --reasoning-parser qwen3 时，输出可能只进入 reasoning 字段而 content 为空。

因此 LLMClient 要做兼容：

content = choice.message.content or ""

reasoning_content = (
    choice.message.reasoning_content
    or choice.message.reasoning
    or choice.message.reasoning_details
    or ""
)

if not content and reasoning_content:
    # 不要直接失败
    # 尝试从 reasoning_content 中提取最后一个 JSON 对象
    content = extract_last_json_object(reasoning_content)

但注意：这是兼容解析，不是让系统依赖模型隐藏思维链。

必改 6：温度和 token 参数

建议 Agent 决策类调用：

temperature: 0.1 ~ 0.3
top_p: 0.8 ~ 0.95
max_tokens: 1024 ~ 2048

不同模块建议：

Preference parsing:
  temperature = 0.1
  max_tokens = 1500

Mission planning:
  temperature = 0.2
  max_tokens = 2000

Decision advisor:
  temperature = 0.2
  max_tokens = 1500

Replanner:
  temperature = 0.2
  max_tokens = 2000

不要一开始就给特别大的 max_tokens。Qwen3.5 系列虽然有很长输出能力，一些页面甚至建议复杂任务可以给很大输出空间，但我们的仿真决策是高频调用，不适合每步长输出。

必改 7：日志字段

每次 LLM 调用都记录：

{
  "model": "qwen/qwen3.5-flash-02-23",
  "provider": "openrouter_or_dashscope",
  "module": "LlmDecisionAdvisor",
  "prompt_tokens": 1234,
  "completion_tokens": 456,
  "reasoning_tokens": null,
  "latency_ms": 812,
  "json_parse_ok": true,
  "selected_candidate_id": "take_order_C123",
  "policy_mode": "budget_tradeoff"
}

不要只看 reason_tokens。

6. 建议执行顺序
第一步

先改模型配置和 LLMClient。

目标：

1. Qwen3.5-Flash 能正常返回
2. JSON 能稳定解析
3. 日志能记录 model、tokens、content、parse result
4. reason_tokens 为 0 也不影响流程

先不要大改策略。

第二步

改 Advisor 输出 schema。

目标：

1. Advisor 不只输出 action
2. Advisor 输出 selected_candidate_id
3. Advisor 输出 policy_mode
4. Advisor 输出 accepted_risks
5. Advisor 输出 rejected_alternatives
第三步

改 CandidateFactBuilder / PlannerScorer。

目标：

1. Planner 不再做 soft risk 过滤
2. 所有候选都带事实
3. soft risk 候选进入 Advisor
第四步

改 CandidateSafetyFilter 和 SafetyGate。

目标：

1. hard invalid 才过滤
2. soft risk 不过滤
3. SafetyGate 不静默改 wait
4. SafetyGate rejection 反馈给下一轮 Advisor
第五步

跑完整仿真并检查日志。

重点看：

1. 是否还有连续 wait
2. Advisor 是否真的看到 soft risk candidates
3. SafetyGate 是否频繁拒绝同类动作
4. Qwen 输出 JSON 是否稳定
5. reason_tokens 是否仍为 0，但不影响决策
6. monthly_income 是否提升
7. preference_penalty 是否失控
7. 最重要的改造红线

后面每次改代码，都用这个标准判断：

如果这段代码是在计算事实，可以保留。
如果这段代码是在替 Agent 做取舍，要收权。
如果这段代码是在保证接口合法，可以保留。
如果这段代码是在根据 rule.kind 写策略，要删除或通用化。
如果这段代码是在处理 hard invalid，可以保留。
如果这段代码是在禁止 soft risk，要交给 LLM。

最终目标不是把代码写得更复杂，而是把权责拆清楚：

Python: 看得清、算得准、拦非法
LLM: 想得全、会取舍、能解释
SafetyGate: 只否决非法，不替你开车
Advisor: 真正决定下一步怎么开
